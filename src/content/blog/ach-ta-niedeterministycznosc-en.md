## How to test LLMs

### Oh, that non-determinism

Language models (LLMs) have one defining characteristic that turns the classic QA approach upside down: they are non-deterministic. This means that even if you ask the same prompt multiple times in the exact same context, the model can generate completely different answers. And it's not because it's malfunctioning - it's because that's exactly how it's designed.

It sounds innocent, but for a tester, it's like verifying a product that behaves differently and unpredictably every single time. In traditional testing, we rely on a fundamental principle: input → expected output. Simple, predictable, and with a defined outcome.
In the LLM world, this rule ceases to exist. Instead of one solid "expected" value, we get a spectrum of possible answers that may be correct... or totally not.
And this is where the uphill battle begins.

### Why does non-determinism break classic tests?

Classic testing assumes determinism; we know exactly what answer to expect. LLMs break this mold, because suddenly:

- "works/doesn't work" stops making sense,
- regression isn't just about comparing two strings,
- "correctness" is no longer binary,
- tests start looking more like grading an essay rather than a hard assertion.

A model might answer perfectly today, "almost perfectly" tomorrow, and wander off into a wild hallucination the day after. It all depends on:

- conversation context,
- prompt phrasing,
- model version,
- temperature and generation parameters,
- or even silent updates made to the model API by the provider.

That's why testing LLMs is no longer about hunting for that one ideal output, but evaluating the quality of the answer against defined criteria.

### What to check when the "expected output" doesn't exist?

Testing language models isn't just about clicking a button and checking if "something popped out." It's still AI, and its answers must be:

- correct (zero hallucinations),
- coherent (no erratic topic jumping),
- compliant with instructions (the model must do what you ask),
- properly formatted (we don't want every answer returning a different JSON schema),
- safe (no toxic content or generation of sensitive personal data).

The QA's job is no longer to check _if_ the model answered, but _how_ it answered.

### Non-determinism is not a problem - it's a challenge

In practice, LLMs can be tested effectively, but it requires a paradigm shift:

- Use **evals** instead of classic assertions.
- Build a **golden set** instead of a single expected output.
- Check by **categories** (accuracy, safety, compliance), not by a single boolean result.
- Remember that the model might drastically change behavior from version to version.
- Use quality grading instead of binary grading.

Non-determinism is not a flaw; it's a feature. And the primary task for QA is to understand how to correctly test it.

### Mini summary: how to think about this in daily QA work

- You don't evaluate if the LLM is "correct" - you evaluate if its answer to a given prompt is correct.
- Instead of a single test, you test quality criteria.
- Instead of expecting an identical string, you expect stability within defined boundaries.
- You verify not just the output, but the risk generated by the answer.
- You check it as a human, not a bot.

### Checklist for this post

- An LLM is never 100% reproducible - that's completely normal.
- Testing must account for variations between answers.
- Output is evaluated qualitatively, not via a 1:1 comparison.
- Non-determinism isn't the bug; ignoring it in your testing process is the bug.
